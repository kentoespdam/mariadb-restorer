## **1\. Executive Summary**

This document outlines the product requirements for a highly resilient, high-performance command-line application designed to restore massive MariaDB/MySQL SQL dump files (e.g., 9GB+). The tool specifically addresses the common pain points of standard GUI tools (crash/OOM) and default CLI commands (slow, lack of resume capability, and dependency issues).

## **2\. Problem Statement**

Restoring large SQL dumps often leads to critical failures:

1. **Out of Memory (OOM):** GUI tools like HeidiSQL crash when loading multi-gigabyte files into RAM.  
2. **View Dependency Failures:** If a View relies on a Table or another View that has not yet been created, the restore fails.  
3. **Circular Table Dependencies:** Tables referencing each other via Foreign Keys cause infinite loops or creation failures.  
4. **Partial Failures:** Network timeouts or power outages during a long restore process leave the database in an inconsistent state. Restarting from scratch takes hours and risks duplicate entry errors.  
5. **Packet Size Limits:** Extended INSERTS can exceed MariaDB's max\_allowed\_packet limit, dropping the connection.  
6. **Slow Performance:** Real-time index building and foreign key checks bottleneck disk I/O, extending a 15-minute job to 14 hours.

## **3\. Product Vision**

To build a lightweight, fast, and crash-resilient CLI tool written in Golang that can reliably parse and restore massive SQL dumps with intelligent dependency resolution, byte-level checkpointing, and extreme speed optimizations.

## **4\. Target Audience**

* System Integrators (specifically for large-scale enterprise deployments like Perumdam).  
* Backend Developers managing Homelab environments (Proxmox, LXC).  
* Database Administrators handling multi-gigabyte migrations.

## **5\. Architectural Blueprint (The "Grilling" Outcomes)**

The application will implement the following core architectural decisions agreed upon during the technical design phase:

### **5.1. Memory Management: Stream Processing**

* **Approach:** Read the .sql file line-by-line using Go's bufio.Scanner (or bufio.Reader for massive single lines).  
* **Benefit:** Memory footprint remains constant (e.g., \~15MB) regardless of whether the file is 1GB or 100GB.

### **5.2. View Dependency Resolution: Dummy Stubbing**

* **Approach:**  
  * As the stream is processed, identify CREATE VIEW statements.  
  * Instead of attempting to resolve complex topological graphs, the app will execute a dummy statement: CREATE TABLE view\_name (dummy INT); immediately.  
  * The actual CREATE VIEW statement is buffered/saved to a temporary queue.  
  * At the very end of the process, execute DROP TABLE view\_name; followed by the actual CREATE VIEW script.  
* **Benefit:** 100% immune to complex or nested view dependencies.

### **5.3. Crash Resilience: Statement-Boundary Checkpointing**

* **Approach:**  
  * Buffer SQL lines in memory until a valid statement terminator (;) is reached.  
  * Execute the statement chunk to MariaDB.  
  * Upon successful execution, atomically save the *byte offset* of the last processed line into a local SQLite tracking database.  
  * **Resume Logic:** If the app crashes, upon restart, it reads the offset from SQLite, uses file.Seek() to instantly jump to the exact byte location, and resumes processing.  
* **Benefit:** Zero data corruption and no need to restart 9GB restores from scratch.

### **5.4. Network Stability: Pre-Flight Parameter Injection**

* **Approach:** Before streaming begins, the app will automatically connect to MariaDB and execute:  
  SET GLOBAL max\_allowed\_packet \= 1073741824; (1GB).  
* **Benefit:** Prevents the "MySQL server has gone away" error caused by massive Extended Inserts statements generated by mariadb-dump.

### **5.5. Ultra-Fast I/O: Session Variable Tuning**

* **Approach:** Upon establishing the database connection for the restore session, execute:  
  SET autocommit \= 0;  
  SET unique\_checks \= 0;  
  SET foreign\_key\_checks \= 0;

  * *Note:* The app must manage explicit COMMIT; commands periodically (e.g., tied to the checkpointing logic) to write data to disk.  
  * Variables are reset at the end of the script.  
* **Benefit:** Bypasses real-time constraint checking, allowing MariaDB to ingest data at maximum disk speed.

## **6\. Technical Stack**

* **Core Language:** Golang (chosen for its superior I/O streaming performance, minimal memory footprint, and single-binary deployment).  
* **Local State Management:** SQLite (embedded via github.com/mattn/go-sqlite3 or a pure Go alternative) for storing byte-offset checkpoints.  
* **Database Driver:** go-sql-driver/mysql for executing commands against the target MariaDB server.

## **7\. MVP Features (Minimum Viable Product)**

1. CLI interface accepting file path, DB host, user, password, and target database.  
2. Stream reader implementation with byte-offset tracking.  
3. SQLite integration for saving/loading checkpoint states.  
4. Regex/String matching to intercept CREATE VIEW and inject Dummy Tables.  
5. Pre-flight DB initialization (disabling checks, increasing packet limits).  
6. Periodic COMMIT execution linked with offset saves.  
7. End-of-stream execution of the intercepted Views.

## **8\. Out of Scope (For MVP)**

* Topological sorting of tables (relying on foreign\_key\_checks \= 0 to bypass circular table dependencies).  
* Handling of Stored Procedures or Triggers (assuming standard table/view structures for now).  
* Concurrent/Parallel streaming (Goroutines Attack) – maintaining linear processing to ensure checkpoint safety.
